Як можна поміряти якість написання коду

Modify Own Code

Кількість рядків коду, змінених автором цього коду не далі ніж три тижні тому.

Modify Own Code Shows the ratio of codelines that were replaced by the same developer (the initial author of the codeline) and that were less than three weeks old.

Refactoring Focus

Кількість рядків коду, що старіші за три тижні та змінених нещодавно.

Refactoring Focus. Shows the Ratio of codelines that were replaced and that were older than 3 weeks.

New Code

Кількість нового коду, написаного в поточному спринті

Shows the ratio of new code the team produced during the current sprint.

Modify Others Code

Кількість рядків коду, змінених не авторами цього коду не далі ніж три тижні тому.

Shows the ratio of codelines that were replaced by another developer (not the initial author of the codeline) and that were less than three weeks old.

Commit Size

Середній розмір коміту в поточному спринті.

Average commit size in the current sprint with commit size as sum of new code, modify own code, refactoring and modify others code

Commit Amount

Середня кількість комітів в поточному спринті

Average amount of commits per engineer and day during the current sprint

Unreviewed PRs

Кількість PR без реакцій та дій зі сторони інших розробників. Оновлення PR новими комітами не рахуються як реакція або дія.

Amount of PRs without any reaction (comments / likes / …) by other developers. Follow up commits do not count as reaction.

Open PR

Кількість відкритих PR

Amount of pull requests that are currently open

PR Cycle Time

Середня кількість часу від відкриття до злиття PR

Avg time from creating a PR till merging a PR

далі розказати про похідні метрики, такі як Monsterfile, Bug Magnet, Bazaar, Supernova і тд

Яка основна задача цих метрик?

  • Це система сповіщення про відхилення від норми, пошук причин — ваша задача
  • Вони не дають розуміння ситуації без контексту
  • Висновки не можна робити без контексту ситуації

Які висновки можна робити з цих метрик?

Цінність для інженера

  • Бачення свого внеску в роботу команди
  • Усвідомлення слабких місць
  • Можливість відслідковувати динаміку своєї роботи в контексті всієї команди

Цінність для керівника

  • Бачення динаміки продуктивності інженера
  • Можливість бачити вплив на продуктивність різних факторів
  • Розуміти як команда працює на індивідуальному та загальному рівні і чому

Висновки

Порозуміння та комунікація!

  • Обговорюйте все незалежно від вашої ролі в команді
  • Дивіться на команду як на одне ціле
  • Як керівнки — загляньте всередину чорної скриньки, почніть розуміти те, що відбувається в команді
